## Hướng dẫn chung

**Luôn đảm bảo sao lưu database trước khi bắt đầu quá trình nâng cấp** bằng cách chạy `docker exec -it {tên_hoặc_id_container_db} pg_dumpall -U {người_dùng_postgres} > databases_backup.sql`.

Để phục hồi sao lưu, chạy `cat databases_backup.sql | docker exec -i {tên_hoặc_id_container_db} psql -U {người_dùng_postgres}`.

Nếu bạn sử dụng Docker Compose, làm theo các bước sau:

1. Trong terminal, trên máy chủ Twenty đang chạy, tắt Twenty: `docker compose down`

2. Nâng cấp phiên bản bằng cách thay đổi giá trị `TAG` trong file .env gần docker-compose. (Chúng tôi khuyến nghị sử dụng phiên bản `major.minor` như `v0.53`)

3. Khởi động lại Twenty với `docker compose up -d`

Nếu bạn muốn nâng cấp instance qua nhiều phiên bản, ví dụ từ v0.33.0 lên v0.35.0, bạn phải nâng cấp instance tuần tự, trong ví dụ này từ v0.33.0 lên v0.34.0, sau đó từ v0.34.0 lên v0.35.0.

**Đảm bảo sau mỗi phiên bản nâng cấp bạn có sao lưu không bị hỏng.**

## v0.60

### Cải thiện hiệu năng
Tất cả tương tác với metadata API đã được tối ưu hóa để có hiệu năng tốt hơn, đặc biệt cho các thao tác metadata đối tượng và tạo workspace.

Chúng tôi đã tái cấu trúc chiến lược cache để ưu tiên cache hit hơn truy vấn database khi có thể, cải thiện đáng kể hiệu năng của các thao tác metadata API.

Nếu bạn gặp bất kỳ vấn đề runtime nào sau khi nâng cấp, bạn có thể cần xóa cache để đảm bảo đồng bộ với các thay đổi mới nhất. Chạy lệnh này trong container twenty-server:

```bash
yarn command:prod cache:flush
```

### v0.55

Nâng cấp instance Twenty để sử dụng image v0.55

Bạn không cần chạy bất kỳ lệnh nào nữa, image mới sẽ tự động xử lý tất cả migration cần thiết.


### Lỗi `User does not have permission`

Nếu bạn gặp lỗi ủy quyền trên hầu hết request sau khi nâng cấp, bạn có thể cần xóa cache để tính lại quyền mới nhất.

Trong container `twenty-server`, chạy:
```bash
yarn command:prod cache:flush
```

Vấn đề này đặc thù cho phiên bản Twenty này và không cần thiết cho các lần nâng cấp tương lai.

### v0.54

Từ phiên bản `0.53`, không cần hành động thủ công.

#### Ngừng sử dụng metadata schema

Chúng tôi đã hợp nhất schema `metadata` vào schema `core` để đơn giản hóa truy xuất dữ liệu từ `TypeORM`.
Chúng tôi đã hợp nhất bước lệnh `migrate` trong lệnh `upgrade`. Chúng tôi không khuyến nghị chạy `migrate` thủ công trong bất kỳ container server/worker nào.

### Từ v0.53

Từ `0.53`, nâng cấp được thực hiện theo chương trình trong `DockerFile`, nghĩa là từ bây giờ bạn không cần chạy bất kỳ lệnh nào thủ công nữa.

Đảm bảo tiếp tục nâng cấp instance tuần tự, không bỏ qua bất kỳ phiên bản chính nào (ví dụ `0.43.3` lên `0.44.0` được phép, nhưng `0.43.1` lên `0.45.0` thì không), nếu không có thể dẫn đến lỗi đồng bộ phiên bản workspace gây ra lỗi runtime và thiếu chức năng.

Để kiểm tra workspace đã di chuyển đúng, bạn có thể xem phiên bản trong database ở bảng `core.workspace`.

Nó phải luôn nằm trong phạm vi phiên bản `major.minor` của instance Twenty hiện tại, bạn có thể xem phiên bản instance trong admin panel (tại `/settings/admin-panel`, truy cập được nếu người dùng có thuộc tính `canAccessFullAdminPanel` đặt là true trong database) hoặc bằng cách chạy `echo $APP_VERSION` trong container `twenty-server`.

Để sửa phiên bản workspace không đồng bộ, bạn sẽ phải nâng cấp từ phiên bản twenty tương ứng theo hướng dẫn nâng cấp liên quan tuần tự cho đến khi đạt phiên bản mong muốn.

#### Xóa `auditLog`

Chúng tôi đã xóa đối tượng chuẩn auditLog, nghĩa là kích thước sao lưu có thể giảm đáng kể sau migration này.

### v0.42.0 lên v0.43.0

Nâng cấp instance Twenty để sử dụng image v0.43.0

```
yarn database:migrate:prod
yarn command:prod upgrade
```

Trong phiên bản này, chúng tôi cũng đã chuyển sang image postgres:16 trong docker-compose.yml.

#### (Tùy chọn 1) Di chuyển database

Giữ image postgres-spilo hiện có là ổn, nhưng bạn sẽ phải cố định phiên bản trong docker-compose.yml là 0.43.0.

#### (Tùy chọn 2) Di chuyển database

Nếu bạn muốn di chuyển database sang image postgres:16 mới, làm theo các bước sau:

1. Dump database từ container postgres-spilo cũ

```
docker exec -it twenty-db-1 sh
pg_dump -U {YOUR_POSTGRES_USER} -d {YOUR_POSTGRES_DB} > databases_backup.sql
exit
docker cp twenty-db-1:/home/postgres/databases_backup.sql .
```

Đảm bảo file dump không rỗng.

2. Nâng cấp docker-compose.yml để sử dụng image postgres:16 như trong file [docker-compose.yml](https://raw.githubusercontent.com/twentyhq/twenty/main/packages/twenty-docker/docker-compose.yml).

3. Phục hồi database sang container postgres:16 mới

```
docker cp databases_backup.sql twenty-db-1:/databases_backup.sql
docker exec -it twenty-db-1 sh
psql -U {YOUR_POSTGRES_USER} -d {YOUR_POSTGRES_DB} -f databases_backup.sql
exit
```

### v0.32.0 lên v0.33.0


### v0.31.0 lên v0.32.0
